OVERVIEW
--------
This document covers all security upgrades applied to the Jomoda platform
across four priority levels (DEFCON 1 through 4). Each section describes
the vulnerability that was found, why it was dangerous, what was changed,
and which file(s) were affected.

All changes are committed to the main branch on GitHub and deployed to Vercel.

1.1  CHECKOUT ACTION USING SERVICE ROLE KEY (Price Manipulation)
-----------------------------------------------------------------
File:    app/actions/checkout.ts
Commit:  f286b66

PROBLEM:
  The checkout server action was using `adminSupabase` (Supabase service role
  key), which bypasses Row Level Security entirely. This meant:
  - Anyone could place orders on behalf of inactive or non-existent vendors.
  - The total price sent from the client was trusted without verification.
    A technically savvy customer could manipulate the cart total to RM 0.01.
  - No input validation existed on any order field.

FIX:
  - Replaced `adminSupabase` with `createClient()` (user-scoped client).
    RLS policies now apply — only active vendors can receive orders.
  - Added full Zod schema validation on all incoming order fields:
      vendor_id (UUID), customer_name (1–100 chars), customer_phone (max 20),
      items (array, min 1, max 50 items), total_price (positive, max 100000),
      delivery_type (enum: pickup | delivery), delivery_address (max 300),
      notes (max 500).
  - Added server-side total re-derivation:
      derivedTotal = sum(item.price × item.quantity)
      If |derivedTotal - clientTotal| > 0.01, the order is rejected.
      The client-submitted total is NEVER written to the database.

KEY CODE (checkout.ts):
  const derivedTotal = payload.items.reduce(
    (sum, item) => sum + item.price * item.quantity, 0
  )
  if (Math.abs(derivedTotal - payload.total_price) > 0.01) {
    return { success: false, error: 'Order total mismatch.' }
  }


1.2  CRON SECRET EXPOSED IN QUERY PARAMETER
--------------------------------------------
File:    app/api/expire-trials/route.ts
Commit:  f286b66

PROBLEM:
  The trial-expiry cron endpoint authenticated via ?secret=CRON_SECRET in the
  URL. Query parameters appear in server access logs, Vercel logs, browser
  history, and referrer headers — the secret was effectively public.

FIX:
  Moved secret to the Authorization request header (Bearer token pattern):
    Authorization: Bearer <CRON_SECRET>

  Any cron job calling this endpoint must now send the header, not a param.

KEY CODE (route.ts):
  const auth = req.headers.get('authorization')
  if (!auth || auth !== `Bearer ${process.env.CRON_SECRET}`) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }


1.3  MISSING HTTP SECURITY HEADERS
------------------------------------
File:    vercel.json (new file)
Commit:  f286b66

PROBLEM:
  The app had no HTTP security headers, leaving it exposed to:
  - Clickjacking (the app could be embedded in an iframe on a malicious site).
  - MIME-type sniffing attacks.
  - Mixed content and protocol downgrade attacks.
  - Overly broad resource loading (no Content Security Policy).

FIX:
  Created vercel.json at project root with the following headers on all routes:

  X-Frame-Options: DENY
    → Prevents the app from being embedded in any iframe.

  X-Content-Type-Options: nosniff
    → Stops browsers from guessing content types.

  Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
    → Forces HTTPS for 2 years, including all subdomains.

  Referrer-Policy: strict-origin-when-cross-origin
    → Limits referrer information sent to external sites.

  Permissions-Policy: camera=(), microphone=(), geolocation=()
    → Disables hardware API access the app does not use.

  Content-Security-Policy:
    default-src 'self';
    script-src 'self' 'unsafe-inline' 'unsafe-eval';
    style-src 'self' 'unsafe-inline';
    img-src 'self' data: https:;
    connect-src 'self' https://*.supabase.co wss://*.supabase.co
                https://fonts.googleapis.com;
    font-src 'self' https://fonts.gstatic.com;
    frame-ancestors 'none';


1.4  SENSITIVE ENV VARS NOT FLAGGED IN VERCEL
----------------------------------------------
Location: Vercel Dashboard → Settings → Environment Variables

PROBLEM:
  SUPABASE_SERVICE_ROLE_KEY and RESEND_API_KEY were stored as plain env vars.
  Vercel's "Sensitive" flag encrypts the value so it cannot be read back
  through the UI even by team members with dashboard access.

FIX:
  Both keys were marked Sensitive in the Vercel dashboard:
  - SUPABASE_SERVICE_ROLE_KEY  → Sensitive: ON
  - RESEND_API_KEY             → Sensitive: ON

  NOTE: This is a dashboard-only change — no code was modified.

2.1  ORDERS RLS POLICY ALLOWED INSERTS TO INACTIVE VENDORS
------------------------------------------------------------
Location: Supabase SQL Editor (manual migration)

PROBLEM:
  Two conflicting INSERT policies existed on the orders table:
  - "Public place order" — correctly checked vendor.is_active = true.
  - "Public can insert orders" — had WITH CHECK (true), no active-vendor check.

  Since Supabase evaluates permissive policies with OR logic, the second policy
  overrode the first. Anyone could insert an order for any vendor_id including
  inactive, suspended, or fabricated vendor UUIDs.

FIX (SQL run in Supabase SQL Editor):
  DROP POLICY IF EXISTS "Public can insert orders" ON orders;

  CREATE POLICY "Public can insert orders"
    ON orders FOR INSERT TO public
    WITH CHECK (
      EXISTS (
        SELECT 1 FROM vendors v
        WHERE v.id = vendor_id AND v.is_active = true
      )
    );


2.2  NO RATE LIMITING ON CHECKOUT
-----------------------------------
File:    app/actions/checkout.ts
Commit:  d4f698a

PROBLEM:
  The checkout action had no rate limiting. A bot could spam thousands of fake
  orders to a vendor, flooding their order dashboard and disrupting operations.

FIX:
  Added a server-side rate limit check before the DB insert:
  - Window: 60 seconds.
  - Limit: 3 orders per customer (identified by phone number, or name if no
    phone provided) per vendor within that window.
  - Uses adminSupabase for the count query (read-only, server-side only —
    this is the only acceptable use of the service role in checkout).
  - If limit exceeded: returns { success: false, error: 'Too many orders...' }

KEY CODE (checkout.ts):
  const windowStart = new Date(Date.now() - 60_000).toISOString()
  const phone = payload.customer_phone.trim()
  const rateLimitQuery = adminSupabase
    .from('orders')
    .select('id', { count: 'exact', head: true })
    .eq('vendor_id', payload.vendor_id)
    .gte('created_at', windowStart)
  const { count: recentCount } = phone
    ? await rateLimitQuery.eq('customer_phone', phone)
    : await rateLimitQuery.eq('customer_name', payload.customer_name.trim())
  if ((recentCount ?? 0) >= 3) {
    return { success: false, error: 'Too many orders. Please wait a minute.' }
  }

NOTE: Login brute-force protection is handled natively by Supabase Auth
(5 failed attempts per 5 minutes per IP). No additional code required.

3.1  NO PHONE NUMBER FORMAT VALIDATION
----------------------------------------
File:    app/dashboard/DashboardClient.tsx
Commit:  db42521

PROBLEM:
  The vendor phone number field (used to generate all WhatsApp links) had no
  format validation. A vendor could accidentally save letters, spaces, or
  symbols, breaking every customer-facing WhatsApp link on their menu page.
  Malicious input could also attempt URI injection via the phone field.

FIX:
  Added client-side validation in ProfileTab.handleSave before any DB write:
  - Strips all non-digit characters.
  - Rejects if resulting digit count is outside 8–15 characters.
  - Saves the digits-only version (e.g. "+60 12-345 6789" → "60123456789").

KEY CODE (DashboardClient.tsx):
  const digitsOnly = phone.replace(/\D/g, '')
  if (digitsOnly.length < 8 || digitsOnly.length > 15) {
    setMessage({ type: 'error', text: 'Phone must be 8–15 digits.' })
    setSaving(false)
    return
  }


3.2  NO URL SCHEME VALIDATION ON IMAGE/LOGO FIELDS
----------------------------------------------------
File:    app/dashboard/DashboardClient.tsx
Commit:  db42521

PROBLEM:
  Vendors could paste any URL into the Logo URL and item Image URL fields,
  including javascript: or data: URIs. While React's JSX rendering prevents
  script injection for most cases, plain <img> tags used elsewhere in the
  codebase do not sanitise src attributes.

FIX:
  Added https:// scheme check in two places:
  1. ProfileTab.handleSave — validates logoUrl before saving.
  2. ItemsTab.handleSubmit — validates form.image_url before saving.
  Any URL not starting with https:// is rejected with an error message.

KEY CODE:
  if (logoUrl.trim() && !logoUrl.trim().startsWith('https://')) {
    setMessage({ type: 'error', text: 'Logo URL must start with https://' })
    setSaving(false)
    return
  }


3.3  ADMIN UPDATE ACTION HAD NO FIELD WHITELIST
------------------------------------------------
File:    app/admin/vendor/[id]/actions.ts
Commit:  db42521

PROBLEM:
  adminUpdateVendor(vendorId, payload) accepted Record<string, unknown> with
  no restriction on which columns could be written. A bug or typo in the admin
  panel could accidentally overwrite sensitive columns such as user_id,
  subscription_status, or created_at.

FIX:
  Added an explicit whitelist of safe vendor columns:

  const ALLOWED_VENDOR_FIELDS = new Set([
    'name', 'slug', 'phone_number', 'is_active', 'is_featured',
    'subscription_status', 'trial_ends_at', 'description', 'promo_text',
    'logo_url', 'gallery_urls', 'payment_methods', 'business_type',
    'blocked_dates',
  ])

  Any key not in this set is silently stripped before the DB update.
  If the resulting payload is empty, the action throws an error.


3.4  NO AUDIT TRAIL FOR ADMIN ACTIONS
---------------------------------------
File:    app/admin/vendor/[id]/actions.ts
Commit:  db42521

PROBLEM:
  All admin write operations (update vendor, add/delete category, upsert/delete
  items) executed silently with no logging. If something was changed or deleted
  incorrectly, there was no way to trace who did what or when.

FIX:
  Added structured console.log to every admin action. These appear in
  Vercel → Project → Functions → Logs in real time.

  Format: [admin] <action> <identifiers> by=<admin-email>

  Examples:
    [admin] updateVendor vendor=abc-123 fields=is_active,name by=admin@jomoda.com
    [admin] deleteItem id=xyz-456 by=admin@jomoda.com
    [admin] addCategory vendor=abc-123 name="Drinks" by=admin@jomoda.com

  The verifyAdmin() helper was updated to return the admin's email address,
  which is then passed to each log line.

4.1  OPEN REDIRECT IN AUTH CALLBACK
-------------------------------------
File:    app/auth/callback/route.ts
Commit:  21fbcb5

PROBLEM:
  The Supabase PKCE callback route accepted a ?next= parameter and redirected
  to ${origin}${next} without validation. A crafted URL like:
    /auth/callback?code=valid&next=//evil.com
  produces:
    https://jomoda.vercel.app//evil.com
  Some browsers treat this as a redirect to evil.com (protocol-relative URL).
  This could be used in phishing attacks — the user sees jomoda.vercel.app in
  the email link but lands on a malicious site after clicking.

FIX:
  Validate next before using it. Only accept values that:
  - Start with / (internal path), AND
  - Do NOT start with // (which would be protocol-relative).
  Anything else falls back to /welcome.

KEY CODE (route.ts):
  const rawNext = searchParams.get('next') ?? '/welcome'
  const next = rawNext.startsWith('/') && !rawNext.startsWith('//')
    ? rawNext
    : '/welcome'


4.2  NO FILE SIZE LIMIT ON UPLOADS
------------------------------------
File:    app/dashboard/DashboardClient.tsx
Commit:  21fbcb5

PROBLEM:
  All four file upload handlers (vendor gallery, item images, room photos, QR
  codes) sent files directly to Supabase Storage with no size check. A vendor
  could upload a 200 MB file, consuming storage quota and causing slow uploads
  that appear to hang.

FIX:
  Added a 5 MB guard at the top of all four handlers before any Supabase call:

  if (file.size > 5 * 1024 * 1024) {
    setError('File too large. Max 5 MB.')
    return
  }

  Affected handlers:
  - GalleryField.handleUpload        (vendor-galleries bucket)
  - ItemsTab.handleImageUpload       (item-images bucket)
  - ItemsTab.handleRoomPhotoUpload   (item-images bucket)
  - SettingsTab.handleQrUpload       (payment-qr bucket)

================================================================================
  KEY ARCHITECTURE RULES (for future development)
================================================================================

  1. NEVER use adminSupabase in a public-facing server action for writes.
     Only createClient() (user-scoped) should perform user-triggered writes.
     adminSupabase is acceptable for: read-only rate-limit checks (server-side),
     admin panel actions (behind verifyAdmin()), and cron jobs.

  2. NEVER trust client-submitted totals or prices.
     Always re-derive monetary values server-side from the canonical source.

  3. ALL new API routes must authenticate via Authorization header,
     not query parameters. Secrets in URLs appear in logs.

  4. ALL new file upload handlers must check file.size <= 5MB before upload.

  5. ALL new redirect handlers must validate the destination is an internal
     path (starts with / but not //).

  6. ALL admin server actions must call verifyAdmin() as the first line
     and log the action with the admin's email.
